# Historia de Usuario: Acceso y redirección basada en roles al panel administrativo

**ID:** `HU-005` | **Prioridad:** `Alta` | **Estimación:** `S`

## Descripción
**Como** usuario autenticado con rol de Administrador (`admin`) o Superadministrador (`suadmin`)  
**Quiero** ser redirigido automáticamente al panel administrativo al iniciar sesión y tener acceso protegido a sus rutas  
**Para** gestionar la plataforma de manera segura y eficiente, evitando que usuarios regulares accedan a estas secciones  

## Criterios de Aceptación

### Escenario 1: Redirección post-login exitosa de Admin/Superadmin
- **Dado** que un usuario inicia sesión con credenciales correctas
- **Cuando** su rol en el perfil de usuario es `admin` o `suadmin`
- **Entonces** el sistema lo redirige automáticamente a la ruta principal del panel administrativo (`/admin/dashboard`)

### Escenario 2: Redirección post-login de usuario regular o estudiante
- **Dado** que un usuario inicia sesión con credenciales correctas
- **Cuando** su rol es `user` o estudiante
- **Entonces** el sistema lo redirige a la landing page o sección pública de cursos (`/courses`) y no le permite el acceso al panel administrativo

### Escenario 3: Bloqueo de acceso no autorizado (Auth Guard)
- **Dado** que un usuario no autenticado o con rol regular intenta acceder directamente a una ruta protegida del panel (ej. `/admin/*`)
- **Cuando** ingresa la URL directamente en la barra de navegación del navegador
- **Entonces** el sistema intercepta la navegación, bloquea el acceso y lo redirige a la página de inicio de sesión (`/login`) mostrando un mensaje de advertencia

### Escenario 4: Validación de sesión mediante token de acceso activo
- **Dado** que un administrador está autenticado y posee un token de acceso activo (`access_token`)
- **Cuando** navega por el panel o realiza cualquier consulta de datos
- **Entonces** el sistema inyecta de manera transparente el token de acceso en la cabecera `Authorization: Bearer <token>` en cada una de las peticiones HTTP dirigidas a la API protegida del servidor.

### Escenario 5: Cierre de sesión por token vencido o inválido
- **Dado** que la sesión del administrador ha expirado o el token de acceso es revocado/inválido
- **Cuando** se realiza una petición HTTP y la API responde con un código de estado `401 Unauthorized`
- **Entonces** el sistema detecta la respuesta de error, destruye la sesión activa en el cliente (limpia el store de Pinia y el `localStorage`) y redirige automáticamente al usuario a la pantalla de `/login` mostrando un mensaje informativo indicando que su sesión ha expirado.

## Notas
- **Manejo del Token de Acceso:** Al iniciar sesión exitosamente, la API debe retornar un `access_token` temporal (JWT o similar) junto a los datos del perfil del usuario. Este token debe persistirse en `localStorage` mediante Pinia para mantener la sesión tras recargar el navegador.
- **Interceptores HTTP:** Se debe configurar un interceptor central en el cliente HTTP (ej. Axios interceptors o custom Fetch wrapper) encargado de:
  1. Adjuntar automáticamente la cabecera `Authorization: Bearer <access_token>` en todas las peticiones salientes.
  2. Interceptar de forma global las respuestas de error `401 Unauthorized` para desencadenar la acción de `logout` limpio y redirección a login.
- **Validación en Navigation Guard:** En el `router.beforeEach`, además de verificar los roles autorizados, se validará la presencia de un token de acceso con formato válido.
- Todas las instalaciones de nuevas librerías y la gestión de dependencias del proyecto se deben realizar obligatoriamente utilizando **pnpm** (ej. `pnpm install`).
